## When to Use This Pattern

**Use flattened HSMs for:**
- Static hierarchies where the structure doesn't change at runtime
- UI flows and wizards with predefined steps
- Simple state serialization needs (string-based state keys)
- Performance-critical scenarios with O(1) event handling

**Use nested HSMs for:**
- Dynamic composition where submachines can be added/removed
- Independent actors that need to run in parallel
- Complex state data that requires nested objects
- Scenarios where event propagation is beneficial

## Comparison

| Feature | Flattened (Flat) | Nested (Propagating) |
| :--- | :--- | :--- |
| **Runtime Structure** | Single Machine | Tree of Machines |
| **State Key** | `"Parent.Child"` | `"Parent"` (Child hidden in data) |
| **Event Handling** | O(1) Lookup | Recursive Bubble/Capture |
| **Serialization** | Simple String | Complex (Recursive Data) |
| **Use Case** | Static hierarchy (UI flows, wizards) | Dynamic composition, independent actors |
