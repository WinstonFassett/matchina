---
title: Frontend Integration Overview
description: How matchina bridges to reactive view libraries. Three patterns cover the common cases.
---

Matchina runs state machines. Rendering is your view library's job. The bridge between them takes one of three shapes depending on what reactive primitive you have on hand.

For build-step React projects, see [React](/matchina/guides/react). For no-build HTML pages (AI-generated artifacts, demos, MDX embeds, progressive enhancement), see [HTML Artifacts](/matchina/guides/html-starters).

## Signal-style: `useMatchina` helper

If your view library exposes a reactive primitive (`signal`, `ref`, `state`), write a four-line helper that mirrors matchina state into the primitive:

```js
const useMatchina = (m) => {
  const s = signal(m.getState());          // or ref / van.state / etc.
  m.subscribe(() => { s.value = m.getState(); });
  return s;
};
```

VanJS, Preact (signals), Solid, and Vue 3 all use this shape.

## Scope-style: mutate on subscribe

If the reactive unit is a plain-object scope (petite-vue, Alpine), skip the helper. The subscribe callback writes directly to scope keys:

```js
mounted() {
  this.machine.subscribe(ev => {
    this.state = ev.to.key;
  });
}
```

## Controller-style: base class

For Hotwire/Stimulus, write one reusable base controller that wires `subscribe` to `render(state)`. Factory machines and `storeApi(store)` both expose transitions as methods, so the action dispatcher just calls `this.api[action](...args)`. The base class is about 20 lines. Each controller after that is short.
